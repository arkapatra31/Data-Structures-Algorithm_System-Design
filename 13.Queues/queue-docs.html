01

Simple (Linear) Queue

The simple queue is the most fundamental queue type. It follows the FIFO (First In, First Out) principle — the element that enters first is the element that leaves first, just like a line of people at a ticket counter.

Internally, two pointers track the state: a front pointer indicating where the next dequeue will occur, and a rear pointer indicating where the next enqueue will place the element. Both start at -1 and increment as operations happen.

Elements enter at the rear and leave from the front — strict FIFO ordering.
Key insight

Once elements are dequeued, the memory slots they occupied at the front of the array become wasted — they can never be reused, even if the array isn't full. This is the fundamental limitation that circular queues solve.

Underlying structure

Array or linked list. Array-based is more cache-friendly; linked list avoids fixed capacity.

Overflow condition

When rear == capacity - 1, even if front slots are empty (phantom overflow).

02

Circular Queue

A circular queue improves upon the simple queue by treating the underlying array as a ring buffer. When the rear pointer reaches the end of the array, it wraps back around to the beginning using modular arithmetic: rear = (rear + 1) % capacity.

This eliminates the "phantom overflow" problem where a linear queue reports itself as full even though freed slots exist at the front. Every slot in the array can be reused, making circular queues far more memory-efficient for fixed-size buffers.

The rear wraps around to reuse freed slots — no wasted memory.
Key insight

A circular queue distinguishes between "full" and "empty" states because both could have front == rear. Common solutions: maintain a count variable, sacrifice one slot as a sentinel, or use a boolean flag.

Full condition

(rear + 1) % capacity == front — the next enqueue position would overlap the front.

Empty condition

front == -1 or a count variable equals zero.

03

Double-Ended Queue (Deque)

A deque (pronounced "deck") generalizes the queue by allowing insertion and deletion at both ends. It can behave as a stack (LIFO) when you only use one end, or as a queue (FIFO) when you insert at one end and remove from the other.

There are two restricted variants: an input-restricted deque allows insertion at only one end but deletion from both, while an output-restricted deque allows deletion from only one end but insertion at both.

Four operations: insert/delete at front, insert/delete at rear.
Key insight

A deque is a superset of both stacks and queues. Python's collections.deque is implemented as a doubly-linked list of fixed-size blocks, giving O(1) operations at both ends and making it the standard choice for BFS, sliding window problems, and any scenario needing fast access at both ends.

As a stack

Insert and delete from the same end (e.g., both from front). Gives LIFO behavior.

As a queue

Insert at rear, delete from front. Gives standard FIFO behavior.

Priority Queue

A priority queue breaks from FIFO ordering entirely. Each element has an associated priority value, and the element with the highest priority (typically the smallest number) is always dequeued first, regardless of insertion order.

The most efficient implementation uses a binary heap — a complete binary tree stored in an array where every parent node has a higher priority than its children. A min-heap serves the smallest value first; a max-heap serves the largest.

Elements are internally organized by priority, not arrival order. The root of the heap always holds the highest-priority item.
Key insight

Insertion triggers a "bubble up" — the new element swaps with its parent until the heap property is restored. Deletion triggers a "bubble down" — the root is replaced by the last element, which then sinks to its correct position. Both operations are O(log n) because the heap's height is log₂(n).

Min-heap vs max-heap

Min-heap: smallest priority served first (common in algorithms). Max-heap: largest value served first (useful for leaderboards, top-K).

Heap as array

For node at index i: left child = 2i+1, right child = 2i+2, parent = (i-1)//2. No pointers needed.

When to Use What

Choosing the right queue type depends on your specific constraints and access patterns. Here's a decision framework:

Use a simple queue when…

You need straightforward FIFO behavior and the queue size is unbounded or you're using a linked-list implementation. Ideal for BFS traversals, task queues where order of arrival matters, and message passing between producer/consumer threads.

Use a circular queue when…

You have a fixed-size buffer and need to maximize memory reuse. Common in embedded systems with limited memory, OS kernel buffers, real-time streaming (audio/video frames), and any scenario where old data can be safely overwritten by new data.

Use a deque when…

You need efficient access at both ends. Critical for sliding window algorithms (finding min/max in a window), implementing both stacks and queues with a single data structure, work-stealing schedulers in parallel computing, and maintaining a recent-history buffer that can be trimmed from either end.

Use a priority queue when…

Elements have varying urgency and the most important item should always be served next. Essential for shortest-path algorithms (Dijkstra, A*), event-driven simulations where events at different times must be processed in chronological order, hospital triage, and any ranking or top-K problem.

Common pitfall

Don't use a priority queue when a simple sort suffices. If you insert all elements before any extraction, sorting the array once — O(n log n) — is simpler and equally fast compared to n insertions into a heap. Priority queues shine when insertions and extractions are interleaved over time.
